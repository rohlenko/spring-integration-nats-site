gws-spring-integration-nats

1. Introduction and Goals

spring-integration-nats is the extension of spring-integration framework developed by Sirius team of FiServ company. It follows the same standards used by other spring-integration-* modules provided by community e.g. like spring-integration-kafka or spring-integration-jms etc. The main goal of this component is to provide integration API in terms and principals of Spring Integration framework, so that the user of this component (developer) will feel himself familiar with it, if he had already experience with the Spring Integration framework and technologies which this framework integrate e.g. like JMS or file based.

  • References

    1. More about NATS technology, link and references please start here https://escmconfluence.1dc.com/display/IPG/NATS+-+Analysis+and+Concept

    2. See in which application the framework has been applied. Please visit References section.

1.1. Requirements Overview

  • Component should be considered as open source and Spring Integration extension and in future can be applied (if accepted by FiServ) to Spring community

  • Should provide DSL API for implementation of any type workflows

  • Should provide integration of NATS technology considering the specific aspects making the technology differ from other message driven technologies

  • Should provide integration of NATS technology and taking in count the power of technology and diversity of area where the technology can be applied. The integration API should consider differ communication models provided by NATS technology.

1.2. Quality Goals

  1. Performance

    • Asynchronous communication:

      1. The solution should use the streaming functionality and full support the power of the streaming approach.

      2. The producers should not be blocked by delivering (sending to the profs - NATS) messaging server. Let the NATS deal with the consumers will let NATS professionally deal with delivery of messages to recipients

    • Stress under the load:

      1. The solution should be resistance to the stress level and provide concept to make it possible

    • Availability & guarantied delivery:

      1. The solution should be considered to be applied in payment domain where guarantied delivery plays important role, that is why the solution should as well consider the low values of availability of messaging server the NATS

    • Exactly-one delivery model:

      1. The solution should be considered to be applied in payment domain where duplicated payment transactions unacceptable from customer point of view. That is why the solution should deal with duplicated messages and worth case minimum report about situation

  2. Monitoring and logging:

    • Monitoring: Each workflow should be easy followed by monitoring solution, where each peace of workflow can be easy explained and troubleshooted using monitoring dashboards

    • Logging:

      1. should let underlined application to select logging implementation it

      2. should produce info logs very thrifty and mostly on debug level. Info logs should be aggregated to one log statement to address thriftiness. An implementing workflow should most take care about logging on info level depending on the need of workflow.

  3. Scalability:

    • Scalable consumers

      1. The solution should provide the ability to easy scale consumers. This will be specially very important in the case when the slowness of the consumers play important role in implementation of business workflow

  4. Backward and Forward compatibility support:

    • Framework should support Backward and Forward compatibilities between producers and consumers. This ability of framework should provide a high level of the flexibility which allows gradual system upgrades without requiring all components to be updated simultaneously. The components of the system are the stakeholders like producers and consumers for NATS, which acts like middleware between these stakeholders. Scenarios can be found in section Backward compatibility scenario. The framework should make possible this software quality.

3.2. Technical Context

The component gws-spring-integration-nats belong to Spring Integration portfolio and is developed by team Sirius in FiServ company. The main aim of this component to provide integration of NATS technology in terms of Spring Integration framework as to the time being of the year 2022 there is no solutions from Spring Integration project to provide the integration of NATS. For more information about Spring Integration project and goals set by community please visit https://spring.io/projects/spring-integration .

The context diagram shows the context where and how the NATS integration component extends Spring Integration framework and provides with the same principals the new nats component (adapters), how spring integration framework does for file based, messaging based over JMS, database based etc. integration approaches. The NATS extension component uses the core functionality of Spring Integration Framework and provides DSL API in the way how new versions of Spring Integration components providing it.

4.3. Quality goals implementation decisions

4.3.1. Decision and solution for Backward and Forward compatibility

Main design decision assumed that the users of nats-spring-integration framework can use the backward compatibility feature out of the box. Using the spring integration flow definition will provide by default the backward compatibility behavior by using default JSON deserializer: Refer to Java Doc API of the MessageConverter.java class to get more detailed information on implementation. The default converter implement the behavior of not failing on unknown property which provides backward compatibility. If for any reasons this functionality should be disabled, the developer has two options. Either use setFailOnUnknownProperties(true) of default converter or use the custom converter in the implementing flow. Please refer to the mentioned JavaDoc for implementation details. Please visit NatsMessageVersioningAddedFieldTest and NatsMessageVersioningDeletedFieldTest for understanding how flow can be implemented.

6.3. Asynchronously message handling

NatsMessageAsyncProducingHandler implements asynchronous publishing method to NATS server (See implementation src/main/java/org/springframework/integration/nats/NatsMessageAsyncProducingHandler.java)

6.3.1. Implementation principals

  1. Separation of main thread responsible for sending messages to NATS server from thread responsible for processing incoming acknowledgment (ack) to the e.g. producer. Such separation guarantees non-blocking behaviour of e.g. producing but from other hand guaranties acknowledgment processing.

  2. Future based processing of acknowledgements by using concepts of CompletableFuture

  3. Buffered separation is based on the Java concept of blocking queue

    • it is required for communication between two threads main ←&→ ack

    • buffered approach helps to mitigate situation with over-performing producers in case like

      1. producers highly over-performing (TODO: good idea to write integration test )

      2. producers over-performing due to under-performing consumers (TODO: good idea to write integration test)

      3. producers over-performing due to slowness or unavailability of NATS server itself (see integration test case in e.g. NatsMessageDrivenChannelAdapterRedeliveryTest)

    • buffered approach helps earlier identify the boundary conditions and provide early feedback like warning or alerting

    • buffered approach helps takes the required measures to mitigate situation created by boundary conditions

6.3.2. Behaviour description

Main message publishing thread

Main thread responsible for messages sending polls incomming messages from dedicated Inbound Channel of spring integration and take care about transformation and preparing for sending to NATS server. Handling of the message is hapanning in the NatsMessageAsyncProducingHandler.java by called from spring integration framework method handleMessageInternaly. The Handler is responsible to check the cpactiy of the buffer implemented like blocking queue and take required measures if requered. If the capacity of buffer reaches threshold value of 75% the main producing process is going to be slow down. The slowdowning is non liniar implemented and depends on the value of buffer capacity. Up to 75% there is no slwodowning and factor is 0. Starting from the function looks like

f(75)= 1 f(80)= 2 f(85)= 4 f(90)= 8 f(95)= 16 f(100)= 32

The Handler in main thread tries to offer the object fo acknowlagement to the blocking queue. In case the capacitiy will reche 100% the slowdowning will have reached maximum value and there message will be just sent to NATS serve and again put to Inbound Chennel. In this case the dupplication of sending messages to NATS server can have a place. But NTAS server as well can handle this situation and provides solution for that. Sending the message to NATS server with message ID NATS will be able to identfy double sent messages and will reject the dupplicates.

Acknowledgement receiver thread

Acknowledgement thread polls messages from blocking queue one by one trying to get acknowlagment. If the acknowledgement from NATS server reciewed in the appropriate time the ack-object will be removed from queue. Otherweise time out can happen and reciving of acknowledgement from Server will be retried until acknowlagment will be recieved by next try. In case for any reason NATS-client can not handle acknowledgement the 3 time such situation will be retried and after reaching 3 re-tries the message will be but into InbountChannel again for resanding to NATS Server. In case of interruptions and cancalation the message will be removed from queue for acknowlagment and resent to InboundChannel again. See sequance diagramm here:

11.2. Quality Scenarios

11.2.1. Backward compatibility scenario

  1. Scenario Situation → Upgraded Producer sends upgraded message content to not upgraded Consumer:

    1. Scenario Type → message content is upgraded with new field

      • It is required upgrade the system where new producer version can send a newer version of message to consumer, where consumer can run with non updated version at this time. In the given scenario the consumer should ignore new content of message and process only known content to the consumer accordingly to the particular consumer version without any impact. As soon consumer will be updated the processing will show new behaviour. This scenario is depicted with following picture:

  1. Scenario Type → message content is upgraded with renamed field

    • This scenario type is not part of the requirement for this framework. This type will be considered exactly like type a. where message content is upgraded with new field. It means, if this type take a place the framework behavior is the ignoring an unknown content and delegate the issue to consumer. It means, that the consumer is responsible for fault tolerance of this scenario type. If the consumer does not care about the handling of such scenario the consumer can fail during the processing of message, if scenario is not considered on consumer side. See Consider best precise and patterns for consumer implementation in case non-supportable scenarios to handle situation on consumer side.

  2. Scenario Type → message content is upgraded with deleted field on producer side.

    • In this scenario type will not be just initialised and will remain null. This means that consumer should care about the field which has not been initialized. A basic problem remains and will be delegated to business logic of consumer. But Framework remains stable! See Consider best precise and patterns for consumer implementation in case non-supportable scenarios to handle situation on consumer side.

11.2.2. Consider best precise and patterns for consumer implementation in case non-supportable scenarios

There are special cases to be considered. They are all about a Breaking Changes. The breaking changes has to be avoided without incrementing the version. If you need such change, do so in a new version to avoid disrupting existing consumers.

  • renaming of field in message

  • deleting of field from message

  • changing semantic of the field from being non-mandatory to becoming a mandatory and vise versa

  • refactoring of message

The best practice is to avoid a renaming or a deleting of message fields. Instead, the recommendation is to mark the fields as deprecated in DTO. Ideal DTO should be designed as centralised library, which both consumer and producer can use. Please see following illustration which shows this concern:

In case of renaming a consumer logic should support both fields during some period of time, which assumes two or more releases. In a dedicated release the deprecated field should be planned for removal.

In case of deleting a consumer logic should be removed in the dedicated release after some dedicated amount of releases.

General speaking changing a non-mandatory field to a mandatory one in an API is often referred to as a "backward incompatible change" or a "breaking change" as well. This is because it has the potential to break existing client applications that rely on the API. Specifically, when you make a non-mandatory field mandatory, you are altering the contract between the API provider and its consumers. This means that existing client applications that do not provide the now mandatory data may no longer function as expected. To handle such changes, it’s generally recommended to introduce a new version of the DTO and clearly communicate the changes to your consumers. This allows existing clients to continue using the old version while new clients can adopt the updated version with the new mandatory field. Concrete speaking, how the framework would behave in this case? This Framework uses jackson library for deserialization. Let imagine the case the producer alters DTO and defines field non-mandatory in V2 which was mandatory in V1. Let assume the upgraded consumer sends the new definition of message to the consumer which still runs with DTO V1. This can be done with jackson library using @JsonProperty(value="someFieldName", required=true) annotation. The same is valid by using @JsonInclude(Include.NON_NULL) or @JsonInclude(Include.NON_EMPTY). The concern is illustrated with following picture:

The functionality will already break in spring integration framework during the deserialization process.

Let us assume another scenario, where producer alters DTO so that a field changes semantic from mandatory to non-mandatory from V1 to V2. OK the deserialization process in framework will not be broken but the problem will be delegated to the business logic of the consumer. It depends on consumer logic of V1 which actually should be implemented so, that it requires this field to process. Very high probable it will fail in this scenario on consumer side. As you can see the shared DTO library does not resolve the issue completely, but it helps to simplify the deprecation, annotation and documentation on code, which in overall simplifies version management.

In case of refactoring it usually means refactoring of producer and consumer business logics itself. In this case there are some options possible

  • a message can be extended with field of version semantic. The consumer should define the behavior on a version

    1. which either can mean that the version can not be supported or

    2. it can mean that the message should be routed to different implementations. See visualised aspect with following illustration:

  • if the message versioning and routing design is not desirable the same can be archived with introduction of new Subject, where new Listener points to new implementation. The versioning is more flexible but more complex concept and should not be underestimated by taking the design decision. On other hand the introduction of new Subject leads to the same as in case old field. This means ols Subject should be supported during some period of time and should be marked as deprecated in the infrastructure and maintenance code. It means double maintenance for Subjects on NATS server. Following illustration makes the designer difference clearer:
